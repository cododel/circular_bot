# Производительность и утилизация CPU

## Проблема: 130–140% CPU при 12 доступных ядрах

При обработке видео в Docker/Dokploy FFmpeg показывает ~130–140% CPU Usage, хотя:
- На хосте доступно 12 ядер
- FFmpeg настроен на `threads=12`
- Ожидается загрузка 200%+ (2+ ядра)

## Причина: архитектура H.264 и межкадровые зависимости

Это **нормальное поведение**, не ошибка конфигурации.

### Почему одно видео не загружает все ядра

**Межкадровые зависимости в H.264:**
- **P-кадры** зависят от предыдущих кадров
- **B-кадры** зависят от предыдущих и следующих
- **Lookahead** анализирует будущие кадры для принятия решений
- Кадр N нельзя закодировать без контекста кадров N-1, N+1

**Практический эффект:**
Распараллеливание есть, но оно:
- Ограничено архитектурой кодека
- Не масштабируется линейно с числом ядер
- Быстро упирается в зависимости между кадрами

### Почему разрешение влияет сильнее длины

Вертикальное видео 720×1280:
- Не является очень тяжёлым по пиксельной нагрузке
- При обработке одним FFmpeg-процессом не создаёт достаточной параллельности
- Стабильно потребляет ~1.3–1.4 ядра даже при доступности большего числа CPU

4K-видео лучше масштабируется, потому что больше блоков = больше работы для motion estimation, transform, deblocking.

## Что означают метрики

### CPU Usage в Docker

| Значение | Значение |
|----------|----------|
| 100% | Полная загрузка 1 логического ядра |
| 140% | ~1.4 ядра |
| 200% | ~2 ядра (при лимите 2 CPU) |

**Важно:** Лимит 2 CPU задаёт верхнюю границу (200%), но не обязывает задачу её достигать.

### threads=12 ≠ загрузка 12 ядер

- `threads=12` — максимальное число рабочих потоков для некоторых стадий
- Не гарантирует одновременную занятость всех потоков
- Часть потоков ожидает: decoder, filter graph, зависимые кадры

## Узкие места в пайплайне

```
[Чтение] → [Декодирование] → [Фильтры] → [Кодирование] → [Запись]
                              ↑
                         Часто bottleneck
```

**Фильтры** (scale, crop, pad, overlay, drawtext) не всегда хорошо распараллеливаются.

## Решение: параллельная обработка нескольких видео

Текущая архитектура не позволяет одному видео загрузить все ядра. Решение — обрабатывать несколько видео одновременно.

### Планируемая реализация (см. Roadmap)

**Очередь обработки с воркерами:**
- Пользователь закидывает пачку кружков
- Каждое видео — отдельный FFmpeg-процесс
- Динамическое распределение CPU на основе утилизации
- Отображение очереди и прогресса каждого видео

**Ожидаемый эффект:**
- 2–4 параллельных процесса = загрузка 200–400% CPU
- Полная утилизация выделенных ресурсов
- Время обработки батча снижается пропорционально числу ядер

## Рекомендации по настройке

### Для текущей версии (одно видео)

```bash
# Оптимизация скорости (жертвуем качеством размытия)
BACKGROUND_BLUR=20        # Уменьшить с 40
FFMPEG_THREADS=4          # Ограничить потоки
```

### Для Docker Compose

```yaml
deploy:
  resources:
    limits:
      cpus: '2'      # Лимит 2 ядра
      memory: 1G
```

Проверка применения лимитов:
```bash
docker inspect <container_id> | grep -E "(NanoCpus|CpuQuota|CpuPeriod)"
```

## Итог

> **Причина не в том, что системе недоступны ядра, а в том, что одиночный пайплайн FFmpeg/H.264 на разрешении 720×1280 не может эффективно занять их все одновременно.**

CPU Usage 130–140% является нормой для одного видео. Для полной утилизации ядер нужна параллельная обработка нескольких видео — запланирована в Roadmap как "Очередь обработки с воркерами".
